# الخطة الشاملة لنشر SHEIN Extractor وربطه بـ n8n وTelegram

## 1. الهدف النهائي

بناء خدمة تعمل دائمًا على Hostinger VPS وتنفذ التسلسل التالي:

```text
رسالة فاتورة نصية في مجموعة Telegram
  -> Telegram Trigger داخل n8n
  -> التحقق من المجموعة والرسالة
  -> إرسال الفاتورة إلى shein-api داخل شبكة Docker
  -> تحليل نص الفاتورة واستخراج رابط SHEIN
  -> تشغيل Chromium عبر Playwright واستخراج المنتجات
  -> إنشاء PDF عربي بالنتائج
  -> إعادة PDF إلى n8n
  -> إرسال PDF ردًا على الرسالة الأصلية في المجموعة نفسها
  -> عند الفشل: إرسال رسالة عربية واضحة بدل الملف
```

البيئة المعتمدة للنسخة الأولى:

- Hostinger VPS بخطة `KVM 2`.
- Ubuntu 24.04.
- Docker وDocker Compose.
- n8n موجود ويعمل حاليًا.
- خدمة Python مستقلة باسم `shein-api` على الخادم نفسه.
- اتصال داخلي بين n8n و`shein-api` دون نشر منفذ API للعامة.

## 2. حدود النسخة الأولى

### ضمن النطاق

- استقبال فاتورة **نصية** من Telegram.
- استخراج اسم العميل ورقم الطلبية ورابط SHEIN من النص.
- رفض النص الذي يحتوي أكثر من رابط SHEIN.
- تحليل رابط واحد في كل طلب.
- استخراج المنتجات المتاحة والنافدة وغير المتوفرة.
- إنشاء PDF عربي متعدد الصفحات.
- إرسال PDF إلى `chat.id` نفسه والرد على `message_id` الأصلي عند دعم إعداد العقدة لذلك.
- إرسال رسالة فشل منظمة عند عدم وجود نتيجة أو عند انتهاء الرابط أو فشل الشبكة.
- الاحتفاظ بنتائج PDF مؤقتًا ثم حذفها تلقائيًا.
- منع تكرار تحليل الرسالة نفسها عند إعادة محاولة Workflow.

### خارج النطاق في النسخة الأولى

- قراءة فاتورة مصورة أو PDF عبر OCR.
- تشغيل واجهة `PySide6` الرسومية على الخادم.
- استقبال أكثر من فاتورة في رسالة واحدة.
- تشغيل عدة عمليات Chromium بالتوازي قبل قياس الموارد.
- بناء تطبيق ويب عام للمستخدمين.
- شراء نطاق منفصل لخدمة `shein-api`.
- Queue خارجي مثل Redis أو RabbitMQ قبل ظهور حاجة فعلية.

إذا أصبح استقبال الصور مطلوبًا لاحقًا، يضاف مسار مستقل: تنزيل الملف من Telegram ثم OCR ثم تمرير النص الناتج إلى نفس طبقة التحليل.

## 3. المعمارية المعتمدة

```text
Internet
  |
  | HTTPS webhook
  v
n8n container
  |
  | Docker private network
  | POST http://shein-api:8000/v1/analyses
  v
shein-api container
  |- FastAPI
  |- invoice_parser.py
  |- extract_cart.py
  |- Playwright + Chromium
  |- PDF renderer
  |- temporary result store
  `- health checks
```

قواعد المعمارية:

1. n8n مسؤول عن Telegram والتفرعات والإشعارات فقط.
2. `shein-api` مسؤول عن منطق الفاتورة والاستخراج وPDF فقط.
3. لا تستخدم محاكاة الضغط على أزرار `app.py`.
4. لا تستخدم عقدة `Execute Command` ولا تمرر نص Telegram إلى shell.
5. لا ينشر منفذ `8000` باستخدام `ports` في Docker Compose.
6. يستخدم n8n اسم الخدمة `shein-api` عبر شبكة Docker داخلية.
7. تبقى واجهة Windows الحالية قابلة للعمل محليًا دون أن تكون جزءًا من الخادم.

## 5. المرحلة صفر: تثبيت القرارات قبل تعديل الكود

### القرارات المعتمدة

- PDF هو صيغة النجاح النهائية.
- JSON يبقى ملفًا داخليًا للتشخيص والسجل، وليس الملف المرسل للعميل.
- نجاح الاستخراج يتطلب `len(products) > 0`.
- المهلة الأولية لتحليل SHEIN: 60 ثانية داخل Playwright.
- مهلة API الكلية المقترحة: 90 ثانية.
- مهلة عقدة n8n المقترحة: 120 إلى 150 ثانية.
- عدد عمليات التحليل المتزامنة في البداية: 1.
- عمر PDF المؤقت: 30 إلى 60 دقيقة.
- المجموعة المسموحة تحدد بقيمة `chat.id` ثابتة.
- يفضل استخدام مجموعة Telegram مخصصة للفواتير فقط.

### نقاط تحتاج اعتمادًا قبل تصميم PDF النهائي

- شعار أو اسم الجهة أعلى التقرير.
- الحقول التي ستظهر في رأس PDF.
- هل تعرض صور المنتجات أم لا.
- ترتيب الأعمدة.
- هل يعرض السعر لكل منتج فقط أم توجد حسابات إضافية.
- النص النهائي لرسائل النجاح والفشل.

يمكن بناء نسخة أولية بتصميم افتراضي ثم تعديل الشكل دون تغيير بقية البنية.

### شرط الخروج

- اعتماد نموذج PDF ومحتوى رسائل Telegram.
- وجود فاتورتين أو ثلاث فواتير نصية للاختبار مع روابط صالحة.
- وجود رابط منتهي أو غير صالح لاختبار الفشل.

### 7.3 خدمة التحليل الموحدة

تنشأ دالة شبيهة بالمفهوم التالي:

```python
analyze_invoice(
    invoice_text: str,
    *,
    timeout_seconds: float,
    request_id: str | None = None,
) -> AnalysisResult
```

مسؤولياتها:

1. تنظيف النص والتحقق من حجمه.
2. استدعاء `parse_invoice_text`.
3. رفض عدم وجود رابط.
4. رفض وجود أكثر من رابط.
5. التحقق من رابط SHEIN.
6. استدعاء `extract_cart(..., headless=True)`.
7. إرفاق اسم العميل ورقم الطلبية ووقت التحليل.
8. اعتبار المنتجات الفارغة `no_products`.
9. حفظ JSON داخلي عند الحاجة.
10. استدعاء مولد PDF.
11. إرجاع نموذج نتيجة موحد.

### 7.4 عدم كسر واجهة Windows

- تعدل `ExtractionWorker` تدريجيًا لاستدعاء الخدمة الموحدة أو تبقى مؤقتًا على المسار الحالي.

## 8. المرحلة الثالثة: توحيد الأخطاء

### 8.1 رموز الأخطاء المقترحة

| الرمز | HTTP | المعنى | رسالة Telegram المقترحة |
|---|---:|---|---|
| `invalid_request` | 400 | الطلب ناقص أو النص غير صالح | تعذر قراءة الفاتورة المرسلة. |
| `invoice_too_large` | 413 | نص أكبر من الحد | نص الفاتورة أكبر من الحد المسموح. |
| `missing_shein_url` | 422 | لا يوجد رابط SHEIN | لم يتم العثور على رابط SHEIN في الفاتورة. |
| `multiple_shein_urls` | 422 | أكثر من رابط | تم العثور على أكثر من رابط؛ أرسل فاتورة واحدة فقط. |
| `invalid_shein_url` | 422 | نطاق أو رابط غير مدعوم | رابط SHEIN غير صالح أو غير مدعوم. |
| `expired_link` | 410 | الرابط منتهي | رابط السلة منتهي أو لم يعد متاحًا. |
| `no_products` | 404 | لا توجد منتجات | لم يتم العثور على منتجات في السلة. |
| `busy` | 429 | الخدمة مشغولة | توجد فاتورة قيد التحليل؛ حاول بعد قليل. |
| `extraction_timeout` | 504 | تجاوز المهلة | استغرق تحليل الرابط وقتًا أطول من المتوقع. |
| `shein_blocked` | 502 | حماية أو حظر | تعذر الوصول إلى بيانات SHEIN من الخادم. |
| `extraction_failed` | 502 | فشل عام في الاستخراج | فشل تحليل السلة، يرجى إعادة المحاولة. |
| `pdf_failed` | 500 | فشل إنشاء PDF | تم تحليل البيانات لكن تعذر إنشاء التقرير. |
| `internal_error` | 500 | خطأ غير متوقع | حدث خطأ داخلي غير متوقع. |

### 8.2 شكل استجابة الفشل

```json
{
  "ok": false,
  "request_id": "...",
  "error": {
    "code": "expired_link",
    "message": "رابط السلة منتهي أو لم يعد متاحًا.",
    "retryable": false
  }
}
```

### 8.3 قواعد التسجيل

- يسجل `request_id` ورمز الخطأ والمدة.
- لا يسجل نص الفاتورة كاملًا.
- لا يسجل Telegram Bot Token أو API key.
- لا يسجل رؤوس المصادقة.
- يمكن تسجيل نطاق الرابط ومعرف المجموعة دون query الحساسة إذا لزم التشخيص.

## 9. المرحلة الرابعة: إنشاء PDF عربي

### 9.1 التقنية المقترحة

- إنشاء HTML/CSS من بيانات `CartExtraction`.
- تحويل HTML إلى PDF باستخدام WeasyPrint.
- استخدام خط عربي واضح مثل `Noto Sans Arabic`.
- تثبيت الخط واعتماديات WeasyPrint داخل Docker بصورة قابلة للتكرار.

هذا الخيار يفصل PDF عن Qt ويجعل تصميم التقرير قابلًا للتعديل عبر HTML/CSS.

### 9.2 محتوى التقرير

رأس التقرير:

- عنوان التقرير.
- اسم العميل.
- رقم الطلبية.
- وقت التحليل والمنطقة الزمنية.
- عدد المنتجات الإجمالي.
- المتاح والنافد وغير المتوفر.
- معرف مجموعة SHEIN عند توفره.

جدول المنتجات:

- التسلسل.
- الصورة أو بديل عند فشل الصورة.
- اسم المنتج.
- SKU.
- الخصائص والمقاس واللون.
- السعر المعروض والعملة.
- حالة التوفر.

تذييل التقرير:

- رقم الصفحة.
- وقت الإنشاء.
- تنبيه مختصر بأن التوفر والسعر يعكسان وقت التحليل.

### 9.3 قواعد PDF

- اتجاه `RTL` للنص العربي.
- أسماء SKU والأرقام يمكن عرضها `LTR` داخل خلاياها.
- دعم عدة صفحات دون قطع الصفوف بصورة غير مقروءة.
- حد أقصى منطقي لحجم الصور ودقتها.
- مهلة مستقلة لتنزيل الصور.
- فشل صورة منتج لا يفشل التقرير كله.
- اسم ملف آمن مثل:

```text
H-2237-محمد-20260722-153000.pdf
```

- تنظيف مكونات الاسم لمنع `/`, `\`, `..` والمحارف غير الصالحة.

### 9.4 اختبارات PDF

- تقرير بمنتج واحد.
- تقرير بعدة صفحات.
- تقرير عربي باسم طويل.
- تقرير دون اسم عميل أو رقم طلبية.
- منتجات بلا صور.
- صورة لا تستجيب.
- أسعار وخصائص فارغة.
- حالات التوفر الثلاث.
- التحقق من أن الملف يبدأ بتوقيع PDF الصحيح ويمكن فتحه.
- Render صفحات PDF إلى صور ومراجعتها بصريًا قبل الاعتماد.

### 9.5 شرط الخروج

- PDF عربي يفتح دون أخطاء.
- لا توجد حروف عربية منفصلة أو معكوسة.
- لا تنقطع صفوف المنتجات بصورة تفسد القراءة.
- الصور الفاشلة تظهر ببديل ولا توقف العملية.

## 10. المرحلة الخامسة: بناء FastAPI

### 10.1 المصادقة

- مفتاح داخلي عشوائي بطول قوي داخل `.env`.
- يرسل n8n المفتاح في رأس مثل `X-API-Key`.
- ترفض الطلبات غير المصرح بها بـ`401` أو `403`.
- رغم أن API داخلية، تبقى المصادقة طبقة حماية إضافية.

### 10.2 نقاط الخدمة

#### فحص الحياة

```text
GET /health/live
```

يتحقق فقط من أن عملية API تعمل.

#### فحص الجاهزية

```text
GET /health/ready
```

يتحقق من إمكانية الكتابة في مجلد النتائج ووجود Chromium واعتماديات PDF.

#### بدء التحليل

```text
POST /v1/analyses
```

مثال الطلب:

```json
{
  "invoice_text": "اسم العميل: ...",
  "source": "telegram",
  "telegram_chat_id": "...",
  "telegram_message_id": "..."
}
```

ويضاف رأس:

```text
Idempotency-Key: telegram:<chat_id>:<message_id>
```

مثال النجاح:

```json
{
  "ok": true,
  "request_id": "...",
  "analysis_id": "...",
  "customer_name": "محمد",
  "order_number": "H-2237",
  "product_count": 17,
  "counts": {
    "normalProducts": 15,
    "outStock": 0,
    "unavailable": 2
  },
  "pdf_path": "/v1/analyses/.../pdf",
  "filename": "H-2237-محمد-20260722-153000.pdf"
}
```

#### تنزيل PDF

```text
GET /v1/analyses/{analysis_id}/pdf
```

- يعيد `Content-Type: application/pdf`.
- يعيد اسم الملف في `Content-Disposition`.
- يتطلب API key.

#### تنظيف النتيجة

```text
DELETE /v1/analyses/{analysis_id}
```

يستدعيه n8n بعد إرسال الملف بنجاح، مع وجود تنظيف دوري احتياطي للنتائج القديمة.

### 10.3 منع التكرار

- يخزن `Idempotency-Key` مع `analysis_id` وحالة الطلب.
- إذا أعاد n8n الطلب نفسه، تعاد النتيجة السابقة بدل فتح Chromium مرة ثانية.
- يمكن استخدام SQLite صغير داخل Volume للنسخة الأولى.
- تنظف سجلات idempotency مع النتائج بعد مدة الاحتفاظ.

### 10.4 التزامن والمهلات

- تشغيل Uvicorn بعامل واحد: `--workers 1`.
- حد التحليل المتزامن: 1.
- يمكن انتظار الطلب التالي في Queue صغيرة أو إرجاع `busy` إذا تجاوز الحد.
- لا يسمح بتكدس غير محدود للطلبات.
- يقتل أو يغلق browser/context في `finally` عند النجاح والفشل.
- تستخدم مهلة تنزيل الصور منفصلة عن مهلة SHEIN.

### 10.5 حدود الإدخال

- حد حجم body على مستوى التطبيق أو reverse proxy إن وجد.
- حد أولي لنص الفاتورة، مثل 32 أو 64 KB، يثبت بالاختبارات.
- السماح فقط بروابط `shein.com` ونطاقاتها المعتمدة.
- منع `file://`, `localhost`, عناوين IP الداخلية، والنطاقات الأخرى لتقليل SSRF.

### 10.6 شرط الخروج

- المصادقة تعمل.
- استجابات الأخطاء ثابتة.
- PDF يمكن تنزيله ثم حذفه.
- تكرار `Idempotency-Key` لا يعيد التحليل.

## 16. المرحلة الحادية عشرة: بناء Workflow في n8n

### 16.1 العقد بالترتيب

1. `Telegram Trigger — Invoice Message`.
2. `Edit Fields — Normalize Telegram Input`.
3. `IF — Allowed Chat`.
4. `IF — Human Text Message`.
5. `IF — Looks Like Invoice`.
6. `Telegram — Processing Message` اختياري.
7. `HTTP Request — Analyze Invoice`.
8. `IF — Analysis Success`.
9. فرع النجاح: `HTTP Request — Download PDF`.
10. فرع النجاح: `Telegram — Send Document`.
11. فرع النجاح: `HTTP Request — Delete Temporary Result`.
12. فرع الفشل: `Edit Fields/Switch — Map Error Message`.
13. فرع الفشل: `Telegram — Send Failure Message`.
14. Workflow منفصل يحتوي `Error Trigger` للأخطاء غير المعالجة.

### 16.2 Normalize Telegram Input

تحفظ حقول ثابتة بأسماء واضحة:

```text
chat_id       = {{$json.message.chat.id}}
message_id    = {{$json.message.message_id}}
user_id       = {{$json.message.from.id}}
is_bot        = {{$json.message.from.is_bot}}
invoice_text  = {{$json.message.text}}
request_key   = telegram:<chat_id>:<message_id>
```

يجب تأكيد شكل بيانات Telegram Trigger الفعلي داخل إصدار n8n الحالي قبل تثبيت expressions.

### 16.3 فلاتر الإدخال

- `chat_id` يساوي المجموعة المعتمدة.
- `is_bot` لا يساوي true.
- `invoice_text` موجود وغير فارغ.
- النص ضمن حد الحجم.
- يمكن التحقق مبدئيًا من وجود `shein.com` لتقليل الطلبات غير الضرورية، مع بقاء التحقق النهائي داخل Python.

الرسائل غير المطابقة يمكن تجاهلها بصمت في المجموعة المخصصة لتجنب الإزعاج.

### 16.4 عقدة Analyze Invoice

الإعداد المقترح:

- Method: `POST`.
- URL: `http://shein-api:8000/v1/analyses`.
- Header: `X-API-Key` من Credential أو قيمة سرية داخل n8n.
- Header: `Idempotency-Key` من `request_key`.
- Body Content Type: JSON.
- Body: `invoice_text`, `source`, `telegram_chat_id`, `telegram_message_id`.
- Timeout: أعلى من مهلة API، مبدئيًا 120–150 ثانية.
- Include Response Headers and Status: مفعّل عند الحاجة للتفرع حسب HTTP status.
- Never Error: يمكن تفعيله كي تصل استجابات 4xx/5xx المنظمة إلى فرع الفشل بدل إيقاف Workflow.

### 16.5 فرع النجاح

النجاح يتطلب:

```text
body.ok == true
AND body.product_count > 0
AND body.pdf_path موجود
```

تنزيل PDF:

- Method: `GET`.
- URL: `http://shein-api:8000{{$json.body.pdf_path}}` وفق شكل output الفعلي.
- Header: `X-API-Key`.
- Response Format: `File`.
- Put Output in Field: `data`.

إرسال Telegram:

- Operation: `Send Document`.
- Chat ID: `chat_id` المحفوظ من Trigger، وليس قيمة ثابتة من عقدة HTTP.
- Binary Data: ملف الحقل `data`.
- Caption: اسم العميل، رقم الطلبية، عدد المنتجات والملخص.
- Reply To Message ID: `message_id` إذا كان الخيار متاحًا في العقدة الحالية.

بعد نجاح الإرسال:

- استدعاء `DELETE /v1/analyses/{analysis_id}`.
- إذا فشل DELETE، لا يفشل إرسال العميل؛ يعتمد cleanup الدوري.

### 16.6 فرع الفشل

- استخراج `error.code` و`error.message`.
- استخدام `Switch` إذا احتجنا نصوصًا مختلفة.
- إرسال رسالة عربية قصيرة ومباشرة.
- إرفاق `request_id` في آخر الرسالة عند الأخطاء الداخلية لتسهيل الدعم.
- عدم إرسال stack trace أو تفاصيل خادم إلى Telegram.

مثال:

```text
تعذر تحليل الفاتورة: رابط السلة منتهي أو لم يعد متاحًا.
رقم المتابعة: 7d3...
```

### 16.7 Error Workflow

ينشأ Workflow منفصل يبدأ بـ`Error Trigger` ويرسل تنبيهًا إداريًا عند:

- توقف عقدة HTTP قبل استجابة منظمة.
- فشل Telegram في إرسال الملف.
- خطأ Credentials.
- تعطل Workflow نفسه.

لا يرسل Error Workflow محتوى الفاتورة كاملًا.

### 16.8 حفظ بيانات التنفيذ

لأن رسائل الفواتير قد تحتوي بيانات عملاء:

- تقليل حفظ successful executions لهذا Workflow إن أمكن.
- عدم Pin بيانات عميل حقيقية داخل العقد.
- تفعيل pruning وعدم الاحتفاظ بالبيانات الثنائية مدة طويلة.
- مراجعة `EXECUTIONS_DATA_PRUNE` و`EXECUTIONS_DATA_MAX_AGE` في n8n.
- استخدام بيانات وهمية عند مشاركة Workflow JSON.

### 16.9 شرط الخروج

- Workflow ينجح يدويًا بفاتورة صالحة.
- فرع الفشل يعمل دون توقف غير معالج.
- PDF يصل إلى نفس المجموعة.
- إعادة تنفيذ execution نفسها لا تكرر التحليل بفضل idempotency.

- إعادة تشغيل VPS في نافذة صيانة.

### 17.5 اختبار الموارد

أثناء 5 إلى 10 تحليلات متتابعة تسجل:

- زمن كل تحليل.
- أعلى CPU.
- أعلى RAM.
- عدد عمليات Chromium بعد النهاية.
- حجم PDF.

- أثر العملية في استجابة n8n.

### 17.6 معايير النجاح

- لا تسقط أي مجموعة توفر من المنتجات.
- كل نجاح ينتج PDF صالحًا.
- كل فشل معروف ينتج رسالة مفهومة.

### 18.1 SSH والخادم

- إنشاء SSH key واختباره قبل تغيير إعدادات الدخول.
- تقليل الاعتماد على root password.
- عدم تعطيل دخول حالي قبل إثبات نجاح المفتاح في جلسة ثانية.
- تحديث حزم الأمان في نافذة صيانة.
- مراجعة `ufw` وHostinger Firewall معًا.
- السماح فقط بالمنافذ اللازمة: SSH وHTTP/HTTPS حسب إعداد n8n.
- عدم فتح Docker daemon أو منفذ 8000.

### 18.2 Docker

- تشغيل `shein-api` كمستخدم غير root.
- `restart: unless-stopped`.
- `init: true` لتقليل zombie processes.
- إعداد Chromium الموصى به داخل Docker، مع توثيق سبب `ipc: host` إذا استخدم.
- تثبيت image والحزم بإصدارات معروفة.
- Log rotation.
- Volume واحد محدد بدل mount واسع من filesystem.
- عدم mount Docker socket داخل `shein-api`.

### 18.3 التطبيق

- API key داخلي.
- Validation صارم للروابط.
- حدود حجم ووقت وتزامن.
- عدم استخدام shell.
- عدم إعادة stack traces للمستخدم.
- عدم تسجيل نصوص الفواتير كاملة.
- توليد معرف طلب لكل عملية.

### 18.4 Telegram وn8n

- Credential منفصل للبوت.
- تقييد Chat ID.
- تقييد User IDs اختياريًا.
- تفعيل 2FA لحساب n8n الإداري.
- عدم مشاركة Workflow يحتوي أسرارًا أو بيانات pinned.
- تقليل مدة الاحتفاظ بالتنفيذات الناجحة والفواتير الثنائية.

## 19. المرحلة الرابعة عشرة: النسخ الاحتياطي والاستعادة

### 19.1 ما يجب نسخه

- بيانات n8n وقاعدة بياناته.
- `N8N_ENCRYPTION_KEY` والأسرار اللازمة للاستعادة في مخزن آمن.
- ملفات Compose.
- `.env` الخاصة بـn8n و`shein-api` في نسخة مشفرة.
- Workflow JSON بعد إزالة البيانات الحساسة.
- كود التطبيق موجود أصلًا في Git.
- لا حاجة عادة لنسخ PDF المؤقتة.

### 19.2 السياسة المقترحة

- Snapshot يدوي قبل كل نشر أو تحديث جوهري.
- الاستمرار في نسخة Hostinger الأسبوعية مبدئيًا.
- تصدير Workflow بعد كل تغيير معتمد.
- اختبار استعادة غير مدمر دوريًا، لا يكفي وجود النسخة دون تجربة.
- توثيق آخر نسخة ناجحة وtag Docker الموافق لها.

### 19.3 سيناريو الاستعادة

1. إيقاف النشر الجديد.
2. إعادة ملف Compose السابق أو image tag السابق.
3. تشغيل n8n وحده والتحقق منه.
4. تشغيل `shein-api` السابقة.
5. اختبار health endpoints.
6. اختبار رسالة Telegram واحدة.
7. إذا تعذر الاسترجاع، استخدام Snapshot Hostinger كخيار أخير لأنه يعيد حالة الخادم كاملة.

## 20. المرحلة الخامسة عشرة: المراقبة والتشغيل اليومي

### 20.1 Health monitoring

- Workflow مجدول داخل n8n يستدعي `/health/ready` كل 5 أو 10 دقائق.
- لا يرسل تنبيهًا من أول فشل عابر؛ يرسل بعد فشلين أو ثلاثة متتالية.
- إرسال تنبيه إداري إلى محادثة خاصة أو مجموعة إدارة منفصلة.

### 20.2 مؤشرات يجب مراقبتها

- CPU وRAM.
- استخدام القرص.
- عدد restarts للحاوية.
- معدل النجاح والفشل.
- متوسط زمن التحليل.
- عدد `expired_link` مقابل `extraction_failed`.
- نمو مجلد النتائج.
- فشل cleanup.

### 20.3 حدود تنبيه أولية

- RAM أعلى من 80% مدة مستمرة.
- Disk أعلى من 75%.
- إعادة تشغيل `shein-api` أكثر من مرة خلال مدة قصيرة.
- ثلاثة أخطاء استخراج متتالية.
- فشل health check مرتين أو ثلاثًا.
- زمن تحليل يتجاوز الحد المعتاد باستمرار.

تعدل الحدود بعد أسبوع من البيانات الفعلية.

### 20.4 الصيانة

- مراجعة logs أسبوعيًا في البداية.
- تنظيف Docker images القديمة بحذر وبعد التحقق من tags المطلوبة للتراجع.
- تحديث Playwright مع Chromium معًا، لا يحدث أحدهما منفردًا.
- أخذ Snapshot قبل تحديث n8n أو صورة المحلل.
- تشغيل اختبارات smoke بعد كل تحديث.
