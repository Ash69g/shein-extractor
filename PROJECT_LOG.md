# سجل مشروع SHEIN Extractor

هذا الملف هو المرجع الرسمي لتقدم المشروع. يوثّق المتطلبات، القرارات، العمليات، الأوامر، الاكتشافات، الاختبارات، المشكلات، والنتائج. يجب تحديثه بعد كل خطوة جوهرية.

## 1. هدف المشروع

بناء برنامج Windows بواجهة عربية يستقبل رابط مشاركة من SHEIN، يحلله، ثم يعرض جميع المنتجات وبياناتها بصورة واضحة. قد يحتوي الرابط منتجًا واحدًا أو عدة منتجات.

### المتطلبات الثابتة

- دعم روابط `onelink.shein.com` الحديثة وروابط المشاركة الوسيطة والنهائية.
- استخراج جميع المنتجات دون استثناء.
- عدم تجاهل المنتجات النافدة أو غير المتوفرة.
- اعتبار قوائم `normalProducts` و`outStock` و`unavailable` أجزاء متساوية الأهمية من السلة.
- دعم السوق السعودي والعملة السعودية مبدئيًا.
- عرض النتائج في واجهة رسومية لاحقًا، مع تأجيل قرار صيغ التصدير النهائية.

### الحقول المطلوبة

- `goods_id`
- `goods_sn`
- `sku_code`
- `goods_name`
- `goods_img`
- `goods_attr` للمقاس واللون والخصائص
- `stock`
- `sold_out`
- `availability`
- `display_price`
- `amountWithSymbol`

## 2. البيئة والأدوات

- نظام التشغيل: Windows.
- محرر الأكواد: Visual Studio Code.
- Python المستخدم في المشروع: `3.14.6`.
- البيئة الافتراضية: `.venv`.
- المتصفح الآلي: Chromium عبر Playwright.
- الحزم الأساسية: `playwright`, `httpx`, `pydantic`, `pytest`, `ruff`.
- إضافات VS Code: Python، Pylance، Python Debugger، Ruff.

### أوامر الإعداد الأساسية

```powershell
py -3.14 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install playwright httpx pydantic pytest ruff
python -m playwright install chromium
```

إذا منعت سياسة PowerShell تفعيل البيئة في جلسة حالية:

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned
.\.venv\Scripts\Activate.ps1
```

## 3. بنية الحل الحالية

1. يستقبل `diagnose_link.py` رابط المشاركة ويشغّل Chromium بمحاكاة هاتف.
2. يتتبع الرابط من `onelink` إلى رابط `appjump` ثم صفحة السلة الفعلية.
3. يسجل طلبات الشبكة والاستجابات والتشخيصات في `diagnostics/`.
4. يراقب `extract_cart.py` استجابة واجهة السلة البرمجية.
5. يجمع المنتجات من مجموعات التوفر الثلاث دون حذف أي مجموعة.
6. يحوّل البيانات إلى النماذج الموحدة الموجودة في `models.py`.
7. يحفظ الناتج بصيغة JSON مؤقتًا داخل `outputs/` إلى أن تُحدد احتياجات التصدير النهائية.

## 4. الاكتشافات التقنية

### مسار الرابط

فتح رابط `onelink` بطلب HTTP عادي لا يكفي دائمًا للوصول إلى صفحة السلة. عند تشغيله في Chromium بمحاكاة هاتف، أصبح المسار كالتالي:

```text
onelink.shein.com
  -> api-shein.shein.com/h5/sharejump/appjump
  -> m.shein.com/ar/cart/share/landing
```

صفحة `appjump` تحتوي معلومات وسيطة مثل نوع المشاركة `cart` ومعرّف المجموعة `group_id`.

### واجهة بيانات السلة

تم تحديد الطلب المسؤول عن بيانات المنتجات:

```text
POST https://m.shein.com/ar/bff-api/order/cart/share/landing?_ver=1.1.8&_lang=ar
```

تحتوي الاستجابة على:

- `allProductSize`: العدد الكلي.
- `normalProducts`: المنتجات المتاحة.
- `outStock`: المنتجات النافدة.
- `unavailable`: المنتجات غير المتوفرة.

### قاعدة تحديد الحالة

مصدر المجموعة له الأولوية على قيمة المخزون. لا يجوز اعتبار المنتج متاحًا لمجرد أن `stock` أكبر من صفر؛ فقد يظهر المنتج داخل `unavailable` مع `stock=999` و` soldOutStatus=true`. لذلك:

- عضو `normalProducts` يصنّف `available`.
- عضو `outStock` يصنّف `out_of_stock` ويضبط `sold_out=true`.
- عضو `unavailable` يصنّف `unavailable` مهما كانت قيمة `stock`.

### قاعدة تمييز المتغيرات

لا يجوز حذف العناصر المتكررة اعتمادًا على `goods_id` أو `sku_code` وحدهما. قد يتكرر المنتج نفسه داخل السلة بمتغيرات مختلفة. وفق التسمية المعتمدة في هذا المشروع، يكون `sku_code` هو الرمز الطويل الذي يبدأ بالحرف `s`، بينما تكون بقية صيغته غير ثابتة. أما `goods_sn` فهو رمز المتغير الذي يبدأ عادةً بالحرف `I`. عند الحاجة إلى مفتاح تمييز، يُستخدم `goods_sn` أو تركيب `goods_id + goods_sn`، مع الاحتفاظ بـ`goods_attr` والمخزون المستقل لكل متغير.

#### قواعد SKU المعتمدة

- التحقق الشكلي الوحيد المطلوب لـ`sku_code` هو أن يبدأ بالحرف `s`.
- لا نفترض صيغة ثابتة لما بعد الحرف الأول؛ قد تكون أحرفًا أو أرقامًا أو اسم فئة مثل `sbbeauty`.
- يجوز تكرار `sku_code` داخل السلة نفسها.
- قد يمثل كل تكرار مقاسًا أو لونًا أو متغيرًا مختلفًا، ويظهر الفرق في `goods_sn` و`goods_attr` والمخزون.
- لا نحذف أي عنصر بسبب تكرار SKU، ولا نستخدم `sku_code` وحده كمفتاح فريد.
- المفتاح الأنسب للمتغير هو `goods_sn`، ويمكن دعمه بـ`goods_id + goods_sn` عند الحاجة.

### تصحيح ربط `sku_code` و`goods_sn`

حقول استجابة SHEIN لا تتوافق مع التسمية المطلوبة للمشروع، ولذلك نعكس الاسمين أثناء التطبيع فقط:

```text
SHEIN goods_sn  -> output sku_code
SHEIN sku_code  -> output goods_sn
```

مثال النتيجة الصحيحة:

```json
{
  "sku_code": "sw260424154768930179807",
  "goods_sn": "I6mocmgbosy2it"
}
```

لا نغير استجابة SHEIN الأصلية؛ التعديل يحدث فقط عند إنشاء نموذج الناتج الموحد.

## 7. مشكلات واجهتنا وحلولها

- كان تثبيت Python 3.13 القديم تالفًا ويعطي `No module named 'encodings'`; تم استخدام Python 3.14.6 وإنشاء البيئة منه بنجاح.
- فشل تفعيل `.venv` أول مرة لأن إنشائها لم يكتمل؛ بعد إنشائها بـ Python 3.14 تم التفعيل بنجاح.
- تعذر تحديث VS Code بسبب بقاء عملية منه مفتوحة ومنع حذف مجلد التحديث؛ الحل إغلاق جميع عمليات VS Code ثم إعادة المحاولة.
- ظهر تحذير `Cache entry deserialization failed`; هو تلف غير حرج في ذاكرة pip المؤقتة وتم تجاهل المدخل.
- ظهر `UnicodeDecodeError` أثناء محاولة قراءة استجابة ثنائية كنص في التشخيص؛ تم التعامل معه دون منع اكتمال التقرير.
- قد تظهر العربية مشوهة في PowerShell القديم رغم أن ملفات JSON محفوظة بصيغة UTF-8 صحيحة.

8. إضافة خدمة `shein-api` إلى شبكة Docker نفسها الخاصة بـn8n، مع `healthcheck` و`restart: unless-stopped` وVolume للنتائج المؤقتة.

#### خانة اسم العميل وتسمية الملف

- يُكتب اسم العميل قبل بدء تحليل الرابط.

2. **الصورة:** صورة مصغرة للمنتج تظهر داخل البرنامج دون فتح المتصفح.
3. **اسم المنتج:** أول 100 حرف فقط لمنع تكدس العرض.
4. **SKU:** القيمة المعتمدة التي تبدأ بالحرف `s`.
5. **الخصائص أو المقاس:** قيمة `goods_attr` كاملة بقدر المساحة المتاحة.

## 2026-07-22 — تحليل تشغيل البرنامج من n8n وTelegram

### ما تم فحصه

- تتبع مسار الفاتورة من `parse_invoice_text` إلى `extract_cart` ثم `write_output`.
- التحقق من أن `extract_cart.py` يملك طبقة استخراج مستقلة يمكن إعادة استخدامها دون تشغيل واجهة `PySide6`.
- التحقق من أن الإخراج الحالي JSON فقط، وأن زري النسخ والتصدير في الواجهة غير مفعّلين، ولا توجد مكتبة أو دالة لتوليد PDF.
- مراجعة وثائق n8n الرسمية لعقد `Telegram Trigger` و`Telegram` و`Execute Command`.

### الخيارات التي جرى تقييمها

1. **خدمة HTTP مستقلة — الخيار الموصى به:** إضافة API تستقبل نص الفاتورة وتعيد PDF أو خطأ منظمًا. يعمل هذا الخيار مع n8n المستضاف ذاتيًا وn8n Cloud، ويفصل Playwright ومكتبات Python عن حاوية n8n.
2. **عقدة `Execute Command`:** مناسبة لتجربة محلية سريعة إذا كان n8n مستضافًا ذاتيًا على الجهاز نفسه، لكنها غير متاحة في n8n Cloud، ومع Docker ينفذ الأمر داخل حاوية n8n لا على الجهاز المضيف. كما أنها معطلة افتراضيًا ابتداءً من n8n 2.0 لأسباب أمنية.
3. **SSH إلى جهاز تشغيل البرنامج:** خيار وسيط إذا كان المستخرج على جهاز منفصل ولا يمكن نشر API، لكنه يزيد إدارة الملفات والمسارات والمفاتيح.
4. **عقدة n8n مخصصة أو إدخال من قاعدة بيانات/طابور:** غير ضروري للنسخة الأولى؛ يصبح مفيدًا فقط عند ارتفاع عدد الطلبات والحاجة إلى عمال متعددين وحالة مهام غير متزامنة.

### التدفق المقترح

```text
Telegram Trigger
  -> التحقق من chat.id ومن وجود message.text
  -> HTTP Request: POST /analyze-invoice
  -> IF success && product_count > 0
       -> Telegram: Send Document إلى chat.id نفسه
     ELSE
       -> Telegram: Send Message برسالة الفشل إلى chat.id نفسه
```

### عقد الخدمة المقترح

- الطلب: `invoice_text` مع معرّفات Telegram لأغراض التتبع فقط عند الحاجة.
- النجاح: PDF كاستجابة ثنائية مع معلومات مختصرة في الرؤوس، أو JSON يحتوي رابط تنزيل مؤقتًا.
- الفشل المتوقع: رمز مناسب ورسالة عربية منظمة مثل `invalid_invoice` أو `multiple_links` أو `expired_link` أو `no_products` أو `extraction_failed`.
- شرط النجاح التجاري: وجود منتج واحد على الأقل؛ فالنسخة الحالية لا تحول قائمة `products` الفارغة إلى فشل تلقائيًا.

### النتيجة

- لا يمكن إكمال السيناريو المطلوب من n8n بالكود الحالي وحده لأن مولد PDF وواجهة الاستدعاء الآلي غير موجودين.
- طبقة الاستخراج الحالية قابلة لإعادة الاستخدام، لذلك لا حاجة لمحاكاة لصق النص أو الضغط على أزرار الواجهة.
- لم يُغيّر الكود ولم تُشغّل الاختبارات في هذه الخطوة لأنها كانت تحليلًا معماريًا وتوثيقًا فقط.

## 2026-07-22 — خيارات استضافة المحلل بجانب n8n على Hostinger

### معلومات البيئة التي تم التحقق منها

- توضح وثائق Hostinger أن قالب `Ubuntu 24.04 with n8n` يشغّل n8n داخل Docker، وأن ملف Compose موجود افتراضيًا في `/root/docker-compose.yml` ويمكن تعديله عبر SSH.

- بحسب صفحة خطط Hostinger وقت التحليل: `KVM 1` يوفّر `1 vCPU / 4 GB RAM`، و`KVM 2` يوفّر `2 vCPU / 8 GB RAM`. الأسعار متغيرة ولم تعتمد كقرار للمشروع.
- توصي وثائق Playwright عند تشغيل Chromium داخل Docker باستخدام `--init` و`--ipc=host` لتقليل العمليات العالقة وتجنب نفاد الذاكرة المشتركة، وبمطابقة إصدار Playwright مع إصدار المتصفح في الصورة.

### الخيار الموصى به للبدء

- تشغيل خدمة مستقلة باسم `shein-api` في حاوية Docker ثانية على **نفس Hostinger VPS** ونفس شبكة Docker الخاصة بـn8n.
- يستدعي n8n الخدمة داخليًا عبر عنوان مثل `http://shein-api:8000/analyze-invoice`؛ وبذلك لا يحتاج منفذ API إلى الظهور على الإنترنت ولا يحتاج نطاقًا عامًا مستقلًا.
- يبقى n8n هو الطرف العام الذي يستقبل Telegram، بينما تنفذ حاوية Python تحليل الفاتورة وPlaywright وتوليد PDF ثم تعيد الملف إلى n8n.
- عدم تشغيل `app.py` أو واجهة `PySide6` على الخادم؛ يعاد استخدام `invoice_parser.py` و`extract_cart.py` والنماذج داخل خدمة API بلا واجهة سطح مكتب.
- بدء التشغيل بتزامن تحليل واحد فقط، لأن كل طلب يفتح Chromium، ثم رفع التزامن بعد قياس فعلي للذاكرة والمعالج.

### تقدير الموارد

- `KVM 1` قد يكفي للتجربة أو الاستخدام الخفيف جدًا، لكنه يترك هامشًا محدودًا عند تشغيل n8n وChromium وتوليد PDF معًا، خصوصًا مع `1 vCPU`.

### البدائل

1. **VPS Hostinger ثانٍ للمحلل:** أفضل عزل واعتمادية، لكنه يضيف تكلفة وإدارة خادم آخر. يتصل به n8n عبر HTTPS ومفتاح API.

- عدم نشر منفذ `8000` للعامة عند استخدام نفس شبكة Docker؛ وإذا اضطر للنشر، يجب وضع HTTPS ومفتاح API وتقييد الوصول.

- `ssh.service` في حالة `active` رغم أن الوحدة نفسها `disabled`، لأن تشغيلها يتم عند الطلب بواسطة `ssh.socket`.
- النتيجة: SSH مهيأ للعودة تلقائيًا بعد إعادة تشغيل VPS ولا توجد حاجة لتفعيل `ssh.service` بصورة منفصلة.
- ما زالت Ubuntu تعرض `System restart required`، لكن إعادة التشغيل مؤجلة حتى التحقق من سياسات إعادة تشغيل حاويتي n8n وTraefik تلقائيًا.

### التحقق من التشغيل التلقائي لـDocker والحاويات

- خدمة Docker في حالتي `active` و`enabled`.
- سياسة إعادة تشغيل الحاوية `root-n8n-1` هي `always`.
- سياسة إعادة تشغيل الحاوية `root-traefik-1` هي `always`.
- الحاويتان تعملان حاليًا؛ Traefik منذ أسبوعين وn8n منذ إعادة تشغيله بعد النسخ الاحتياطي.
- أصبحت مكونات الوصول والخدمات الأساسية مهيأة للعودة تلقائيًا بعد إعادة إقلاع VPS: `ssh.socket` وDocker وn8n وTraefik.
- قبل تثبيت التحديثات وإعادة التشغيل، تقرر فحص قائمة الحزم القابلة للترقية وسجل فشل `unattended-upgrades` والوحدات الفاشلة لتجنب تحديث أعمى.

### نتيجة فحص تحديثات Ubuntu قبل الترقية

- لا توجد وحدات systemd فاشلة؛ أعاد `systemctl --failed` عددًا قدره `0`.
- إعادة التشغيل مطلوبة لتفعيل `linux-image-6.8.0-136-generic` وتحديث `linux-base`، بينما النواة الحالية في الجرد السابق كانت `6.8.0-134-generic`.
- قائمة الترقيات المتاحة تشمل مكونات Ubuntu الأساسية والشبكة وAppArmor وcloud-init، وتشمل كذلك Docker وcontainerd وDocker Compose.
- سجل `unattended-upgrades` لا يعرض فشل تثبيت فعلي في آخر تشغيل؛ بل سجل `All upgrades installed` بتاريخ 2026-07-22، مع إبقاء `cloud-init` مؤجلًا/held back.
- الاستنتاج: رسالة وجود تحديث لم يُثبت تلقائيًا ترتبط على الأرجح بالحزمة المؤجلة `cloud-init` وليست عطلًا في خدمة قائمة.
- نظرًا إلى وجود تحديثات Docker ضمن القائمة، لن تنفذ ترقية مباشرة. الخطوة التالية هي تحديث فهارس APT ثم تشغيل محاكاة `apt-get -s upgrade` ومراجعة الحزم التي ستثبت أو تزال قبل الموافقة على التنفيذ الفعلي.

### نتيجة محاكاة ترقية الخادم

- نجح `apt-get update` في تحديث الفهارس من مرايا Ubuntu الرسمية ومستودع Docker دون أخطاء ظاهرة.
- أظهرت محاكاة `apt-get -s upgrade` عدد `55` عملية تثبيت/ترقية، وعدد `0` عمليات إزالة.
- لا توجد حزم سيحذفها APT وفق المحاكاة.
- تشمل التغييرات المهمة Docker CE من `29.0.2` إلى `29.6.2`، وcontainerd من `2.1.5` إلى `2.2.6`، وDocker Compose plugin من `2.40.3` إلى `5.3.1`، إضافة إلى netplan و`linux-base`.
- لم تظهر `cloud-init` ضمن الحزم المهمة التي ستنفذها ترقية `upgrade` الحالية، وهو متوافق مع كونه held back.
- القرار: تنفيذ `apt-get upgrade` بصورة غير تفاعلية مع الاحتفاظ بملفات الإعداد الحالية عبر `--force-confold`، وحفظ السجل في `/root`، ثم فحص Docker وn8n والخدمات قبل تنفيذ أي إعادة تشغيل للخادم.

### نتيجة الترقية الفعلية للخادم قبل النشر

- اكتمل أمر `apt-get upgrade` بنجاح وأعاد رمز الخروج `0`، وحُفظ سجله في `/root/predeploy-upgrade-20260722-075138.log`.
- لم يعرض `dpkg --audit` أي حزم ناقصة الإعداد أو أخطاء في قاعدة الحزم.
- تمت ترقية Docker إلى `29.6.2` وDocker Compose إلى `5.3.1`.
- عادت حاويتا `root-traefik-1` و`root-n8n-1` للعمل بعد الترقية، وأظهر فحص n8n الصحي النتيجة `{"status":"ok"}`.
- جرى إنشاء ملف الإقلاع للنواة الجديدة `6.8.0-136-generic`، بينما ما تزال النواة المشغلة `6.8.0-134-generic`؛ لذلك يلزم إعادة تشغيل VPS لتفعيل النواة الجديدة.
- ظهرت وحدة واحدة فاشلة هي `fwupd.service` أثناء إعادة تشغيل الخدمات. هذه الخدمة خاصة بتحديث Firmware وليست جزءًا من Docker أو Traefik أو n8n، لكن سيعاد فحصها بعد إعادة تشغيل الخادم قبل تجاهلها أو تعطيلها.
- لم تُنفذ إعادة التشغيل بعد. الخطوة التالية هي إعادة تشغيل VPS بشكل مضبوط، ثم التحقق من النواة وSSH وDocker والحاويات وصحة n8n قبل بدء نشر خدمة SHEIN.

### إعادة تشغيل VPS وتفعيل النواة الجديدة

- أُعيد تشغيل VPS بعد اكتمال الترقية، وظهرت شاشة الدخول إلى `Ubuntu 24.04.4 LTS` بالنواة الجديدة `6.8.0-136-generic`، ما يثبت نجاح تفعيل تحديث النواة.
- بعد عودة جلسة Terminal كُتب أمر `reboot` مرة أخرى، ولذلك ظهرت الرسالة `Your session ended` ونُفذت إعادة تشغيل ثانية. هذا لا يدل على عطل ولا يسبب ضررًا للخادم، لكنه يتطلب انتظار عودة VPS وفتح Terminal مرة أخرى.
- الخطوة التالية: فتح جلسة Terminal جديدة دون تكرار `reboot`، ثم تنفيذ فحوص ما بعد الإقلاع لـSSH وDocker وTraefik وn8n و`fwupd.service`.

### نتيجة فحوص ما بعد إعادة الإقلاع

- يعمل VPS بالنواة الجديدة `6.8.0-136-generic`، وكان زمن التشغيل عند الفحص دقيقتين.
- `ssh.socket` في حالتي `active` و`enabled`، ما يؤكد عودة الوصول إلى Terminal بصورة صحيحة.
- أعاد `systemctl --failed` العدد `0`، ولا توجد أي خدمة فاشلة بعد إعادة الإقلاع.
- خدمة Docker في حالتي `active` و`enabled`، والإصدارات الحالية هي Docker `29.6.2` وDocker Compose `5.3.1`.
- عادت حاويتا `root-traefik-1` و`root-n8n-1` تلقائيًا، وكانتا في حالة `Up 2 minutes` وقت الفحص.
- أعاد فحص n8n الصحي `{"status":"ok"}`.
- أصبحت `fwupd.service` في حالة `inactive (dead)` كوحدة `static` وليست في حالة فشل؛ وبذلك زال فشلها السابق بعد إعادة الإقلاع ولا يلزم إصلاحها.
- أصبحت ترقية النظام وإعادة الإقلاع والتحقق من الخدمات الأساسية مكتملة بنجاح. الخطوة التالية هي فحص أسماء التحديثات الثلاثة المتبقية وسبب رسالة التحديث التلقائي قبل متابعة إعداد النشر.

### مراجعة التحديثات المتبقية بعد الإقلاع

- أظهر `apt list --upgradable` حزمتين فقط: `cloud-init` من `24.1.3-0ubuntu3` إلى `26.1-0ubuntu1~24.04.1`، و`fwupd` من `1.9.31-0ubuntu1~24.04.1` إلى `2.0.20-1ubuntu2~24.04.2`.
- يوضح سجل `unattended-upgrades` أن آخر دفعة تحديثات تلقائية اكتملت برسالة `All upgrades installed`، ولا يظهر فشل في `dpkg`.
- رسالة عدم تثبيت تحديث تلقائي ترتبط بحزمة `cloud-init` التي يسجل النظام صراحة أنها `marked to be held back`، وليست بسبب عطل في مدير الحزم.
- لا تُثبت `cloud-init` يدويًا ولا يُلغى حجزها حاليًا لأنها مرتبطة بتهيئة VPS لدى مزود الاستضافة، والخادم يعمل بصورة سليمة.
- لا يلزم تحديث `fwupd` قبل النشر لأنها خدمة Firmware غير مستخدمة فعليًا داخل VPS، وهي حاليًا `inactive (dead)` وليست فاشلة.
- القرار: إبقاء الحزمتين دون تغيير، واعتبار مرحلة تحديث VPS والتحقق منه مكتملة. المرحلة التالية هي تهيئة مشروع SHEIN للعمل كخدمة خلفية headless متكاملة مع n8n.
